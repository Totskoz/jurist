# M5 eval suite — pre (marker)

**Status:** Not captured as a suite run.

`scripts/eval_suite.py` was written as part of M5 (Task 18), so it did not exist
on the M4 tip (`c6833d4`). There is no "pre-M5 suite run" to compare against —
the harness postdates the baseline.

## What serves as the pre-M5 baseline

The qualitative evidence that motivated M5 lives here:

- **`docs/evaluations/2026-04-22-m4-e2e-run.md`** — two real-LLM runs of the
  locked huurrecht question on `master @ aeb8282`. Event counts, timings, and
  the rendered Dutch answer.
- **`docs/superpowers/specs/2026-04-22-m5-answer-quality-design.md` §2** —
  external-review findings AQ1–AQ8 that drive the M5 scope. These findings
  *are* the pre-M5 baseline: each one enumerates a concrete M4 miss on the
  same demo question.

The M5 post run (`2026-04-22-m5-suite-post.md`) is therefore compared against
those enumerated findings, not against a prior suite output.

## Why we did not retro-run a pre baseline

Reconstructing a pre-M5 suite run would require checking out `c6833d4`,
porting the harness back (it does not exist at that SHA), re-indexing cases
without the five M5 fence terms, and re-running all five questions against
live LLMs. The cost is roughly double Task 25's real-API spend for a metric
whose value we already know qualitatively — we have the M4 trace, we have the
review findings, and the questions in the suite were chosen specifically to
exercise those findings. A synthetic pre number would at best reproduce what
we already know.

If a future milestone wants a true before/after, the harness is now in place
and can capture a pre baseline *before* the next round of changes.
