# M4 end-to-end evaluation — 2026-04-22

Full pipeline run against the real Anthropic API on the locked demo question.
Two back-to-back runs, same code (`master @ aeb8282`), same question, same
ingested corpus — captured via `scripts/eval_m4_run.py` (buffer raised to 10k
to avoid history eviction; see "Limitation L1" below).

Question:
> *Mijn verhuurder wil de huur met 15% verhogen per volgend jaar, mag dat?*

Output artefacts live under `out/m4-eval/`:
- `trace.jsonl` — every stamped `TraceEvent`
- `summary.json` — timings, event counts, final-answer shape
- `answer.md` — rendered Dutch answer for human inspection

## 1. Pipeline behaviour (run #2 — happy path)

| Stage | Wall | Events emitted | Notes |
|---|---|---|---|
| `run_started` | 0.0s | 1 | — |
| decomposer | 2.9s | 2 | one Haiku forced-tool call; no thinking events |
| statute_retriever | 35.9s | 58 | 3-turn Sonnet tool-use loop; 10 calls (9 `get_article` + 1 `done`); 9 `node_visited`; 27 `agent_thinking` tokens; 7 articles returned to synth |
| case_retriever | 5.5s | 23 | embed + LanceDB top-150 → 20 unique ECLIs → Haiku rerank to 3; `reranked` fires once |

| **total** | **97.8 s** | **720** | — |

The orchestrator's async boundary between stages is clean — every agent
finishes before the next starts (no concurrency, as spec §15 intended for
v1). Terminal event is `run_finished`; validator reports `valid=true`.

## 2. Answer quality (human read — non-lawyer but Dutch-competent)

The full answer is in `out/m4-eval/answer.md`. Substantively:

**Korte conclusie** (4 sentences) — correctly states that 15 % is in
beginsel niet toegestaan, names the CPI + 1 procentpunt mechanism, notes
partial nietigheid. Legally accurate for both geliberaliseerde and
geregulateerde huur post the 1 mei 2021 reform.

**Wetsartikelen** — Uhw 10 (the anchor), BW 7:248 (nietigheid-beding), BW
7:265 (dwingend recht), BW 7:252 (procedurevoorwaarden). Quote for Uhw 10
is long (382 chars) but within the 40-500 bound and carries the full
formula. Each `explanation` ties the quote to the 15 % question explicitly
— not generic restatement.

**Uitspraken** — one recent appellate hof (GHDHA 2025), two rechtbank
decisions, all reasoning directly about CPI + 1 procentpunt as a cap or
the twee-grondslagen structure of 7:248. Very on-topic.

**Aanbeveling** — actionable: (1) check contract for beding, (2) verify
notification formalities, (3) schriftelijk bezwaar vóór ingangsdatum,
(4) Huurcommissie-verzoek binnen 4 maanden na ingang. Matches the
procedural chain 7:252 → 7:253.

**One OCR artefact in the source** surfaced in the first case quote:
> "de jaarlijkse huurverhoging wordt gemaximeerd op de inflatie plus **l** procentpunt"

The `l` is a `1` mis-recognised during rechtspraak.nl's XML generation.
The synthesizer correctly quoted it verbatim (grounding depends on
character-exact match), so the artefact lands in the final answer. This
is a **corpus data issue**, not a pipeline bug — worth noting because a
lawyer-user would scan for this and draw a wrong conclusion about
pipeline quality. See Improvement I5.

## 3. What the pipeline does well

### G1 — Closed-set grounding held end-to-end
Every citation survived the `verify_citations` strict-substring check
(NFC + whitespace-collapse) against the article `body_text` and case
`chunk_text` surfaces. No hallucinated `article_id` / `bwb_id` / `ecli`.
The three-layer defense (JSON-Schema `enum` → Pydantic → substring)
means we catch hallucination at the cheapest possible layer in practice
and fall back to post-hoc only when the model cheats (see below).

### G2 — The regen path actually fires
Run #1 (not shown in the table above — numbers lost when the 500-event
buffer evicted early history, see L1) logged:
> `synthesizer attempt 1 invalid (schema_ok=True, failures=1) — retrying once`
>
> → final terminal: `run_finished`

One citation failed substring verification on attempt 1; the Dutch
advisory was appended to the user message; attempt 2 produced citations
that verified cleanly. This is exactly the fallback path the M4 spec
specified, and it worked silently on real traffic — the user doesn't see
the failure. The regen doubled synthesizer latency on that run
(137 s vs 53 s for a single-attempt run) but stayed below the 15-iter /
90 s caps (those only apply to the statute retriever).

### G3 — Statute retriever is efficient on this question
Sonnet batched tool calls: turn 1 issued 7 parallel `get_article` calls,
turn 2 issued 3 more, turn 3 thought for 24 s then called `done`. That's
three round-trips for a 9-article exploration — none of the coercion
paths (`dup_loop`, `stall`, `done_error`, `max_iter`, `wall_clock`)
triggered. The system-prompt catalog + cache_control breakpoint are
doing their job: the model can pick nodes directly without a
`search_articles` call.

### G4 — Case rerank gives meaningful ranking
Top-20 cosine candidates landed in a tight similarity band (0.71 or so
for the top three). Haiku's rerank picked a structurally diverse set:
one Hof ruling (2025, most recent), two rechtbanken, each reasoning
about a different facet of the problem (CPI-cap, oneerlijk beding,
twee grondslagen). Picking from raw cosine alone would have been a
different 3 and possibly duplicative — rerank earns its Haiku call.

### G5 — Event stream genuinely reflects work
The 720 events cover every decision the pipeline makes: which articles
got inspected (`node_visited` × 9), which tools called (`tool_call_*`
× 10), which cases surfaced (`case_found` × 20), which 3 were kept
(`reranked` × 1), which citations resolved to a URL (`citation_resolved`
× 7), and the synthesized answer streamed word-by-word (`answer_delta`
× 623). A UX built on this has real diagnostic power — you can point at
any entity in the final answer and trace backwards to the agent decision
that put it there.

## 4. Limitations

### L1 — Default `EventBuffer(max_history=500)` silently drops early events
The first evaluation run produced 500 events in the drained buffer
(same as the cap), so I initially believed the pipeline emitted ~500.
The second run with `max_history=10_000` showed the real total is **720
events** — every event up to index ~220 had been evicted. Since the
SSE endpoint drains via `EventBuffer.subscribe()`, a late subscriber
on the API today sees the same eviction: everything decomposer /
statute / case_retriever emits before the synthesizer starts its word
replay is at risk once the answer is long enough.

CLAUDE.md's current claim ("Full run emits ~250+ events post-M4") is a
material undercount for anything but very short answers. On this run,
the 623 `answer_delta` events alone exceed the 500 cap; on a longer
answer (e.g. a multi-part question) the ceiling is hit harder.

**Why it matters:** the frontend's TracePanel replays from this history.
A user who switches away and back mid-run loses the early agent
evidence — which is exactly the evidence the demo is meant to show.

### L2 — The synthesizer ignores "think out loud first"
The system prompt (`src/jurist/llm/prompts/synthesizer.system.md`)
explicitly asks for 1-3 sentences of Dutch pre-tool reasoning:
> "Denk eerst kort hardop in het Nederlands over welke bronnen je gaat
> citeren en waarom."

On this run the synthesizer emitted **zero** `agent_thinking` events
from Sonnet. CLAUDE.md advertises "typically 5-20 events depending on
prompt adherence" — that's aspirational, not descriptive. With
`tool_choice={"type":"tool","name":"emit_answer"}` (forced tool),
Sonnet generally skips the freetext preamble and goes straight to the
tool call. The instruction in the prompt has no teeth.

**Why it matters:** the UX contract says the AnswerPanel fills in from
`answer_delta` while TracePanel shows parallel "agent thinking" from
the synthesizer. In practice the trace panel for the synthesizer is
blank until citations resolve — an empty box for 34 seconds.

### L3 — Statute retriever system prompt is English; initial thinking leaks English
The `statute_retriever.system.md` template is in English and only
pseudo-instructs the model to reason in Dutch:
> "Policies / Reason in Dutch when considering article content."

On this run, the first 3 `agent_thinking` events were in English:
`"I"`, `"'ll analyze the catalog..."`, `"Let me load the key articles..."`
— only later (after the first tool-use turn) did the reasoning shift
to Dutch. In a Dutch-language product, English in the live trace panel
is a clear quality tell.

### L4 — Wall-time variability is high (53 s → 137 s on the synthesizer)
Two runs on the same question, same code:
- Run A: 181.8 s total, synthesizer ~137 s (regen fired once)
- Run B: 97.8 s total, synthesizer ~53 s (no regen)

A ~2× spread on the dominant stage is driven entirely by whether the
first synthesizer attempt happens to produce a citation that fails
verification. There's no cache on the second attempt (`ephemeral`
cache_control only benefits the system prompt, not the user message
with the appended advisory), so the regen pays full latency again.
An interview demo where the question takes 3 minutes when it might have
taken 1.5 minutes is an uncomfortable surprise.

### L5 — Per-run latency is dominated by the synthesizer replay, not the LLM
Of the synthesizer's 53.5 s: ~34 s is the actual Sonnet streaming call
that produces the validated `StructuredAnswer`, and ~20 s is the
synthetic word-level replay (`_TOKEN_SLEEP_S = 0.02` × ~623 words +
scheduler overhead). The replay was a UX-preservation choice — M4
removed the free-text streaming but wanted to keep the AnswerPanel
"typing" effect — but it adds non-trivial wall time to every
successful run. The buffer sees 623 events; the network sees all 623
frames with pauses between them.

### L6 — `cited_articles` body sizes are unbounded
The statute retriever returned 7 articles with body sizes 113 / 816 /
1164 / 1993 / 3465 / 4538 / 4545 chars. Those full bodies go into the
synthesizer's user message verbatim (spec §5 requires it for
grounding). On a query that loads `7:252a` + `7:253` (~8000 chars by
themselves) plus a dozen case chunks, the input can get big. No
specific cap today — cost scales linearly.

### L7 — Decomposer concepts drift slightly off-target
The concepts list included `"Wet op de huurtoeslag"` (rent subsidy law,
not directly on-topic) and `"prijzenbinding"` (neologism, no legal
referent). These don't affect downstream retrieval — statute and case
retrievers use the sub-questions, not the concepts — but they do show
in the trace panel if it surfaces them, and a reviewer will notice.

### L8 — Permanent validator stub is a live gap
`validator_stub` always returns `valid=True`. The orchestrator has no
post-synthesis check beyond the citation-grounding verification already
run inside the synthesizer. The stub is explicitly a v2 item in the
spec, but on the current branch it doesn't catch things like: duplicate
`article_id`s in `relevante_wetsartikelen`, `article_label` that
disagrees with `article_id`, or an `aanbeveling` that contradicts the
`korte_conclusie`.

### L9 — OCR / ingest artefacts pass through unchanged
The `l` / `1` confusion in the Hof Den Haag 2025 quote
("inflatie plus l procentpunt") comes from rechtspraak.nl's XML. The
pipeline has no OCR-repair step, and wouldn't want one at quote time
(grounding needs character-exact match). But the artefact lands in a
user-facing answer. Fixing this belongs in M3a ingest, not the
agents — see I5.

## 5. Improvements

Ordered by value × cost. Each is scoped to what it alone would deliver;
none requires a full architectural revisit.

### I1 — Raise default `max_history` on `EventBuffer`
Cheapest fix, biggest UX win. `src/jurist/api/sse.py:27` currently
`max_history: int = 500`. Bump to `2000` (covers ~3× the observed run),
keep `max_history_per_run` in `settings` in sync. Cost: two integer
changes. Won't grow memory meaningfully — 2000 TraceEvents is
~500 KB per in-flight run.

### I2 — Drop pre-tool thinking requirement from synthesizer prompt, or replace with real streaming content
The forced-tool constraint means Sonnet won't emit pre-tool thinking
on most turns. Two options:
- **Option A (honest):** remove the "Denk eerst kort hardop" block from
  `synthesizer.system.md`, update CLAUDE.md to say the synthesizer
  emits 0 thinking events in the happy path, and rewrite frontend copy
  so the trace panel for `synthesizer` shows something other than
  "waiting for thinking…" while the tool call is in flight.
- **Option B (tighter):** replace the forced-tool with `tool_choice:
  {"type":"auto"}` plus an explicit system-prompt requirement that the
  model "first write 1-3 sentences, then call `emit_answer`". This
  costs nothing if the model complies, but risks the model skipping
  the tool call and requiring regen logic to handle a no-tool path.

My recommendation: **A**. The value of synthesizer pre-tool thinking
is debatable — the decomposer and retrievers already show the
reasoning chain. The synthesizer thinking would mostly restate picks.

### I3 — Translate statute_retriever system prompt to Dutch
`statute_retriever.system.md` in Dutch. This is maybe 40 lines of
localisation, plus a regression check on the existing unit tests (the
fake MockAnthropicClient doesn't care about language; the live tests
test behaviour, not language). CLAUDE.md's "Reason in Dutch" is
already project policy for agents facing Dutch users — statute
retriever is the outlier.

### I4 — Cap synthesizer regen latency penalty via partial input reuse
The current regen constructs `user_retry = user + "\n\n" + advisory`
and makes a fresh streaming call. Sonnet re-reads the entire article
bodies + case chunks, which is what dominates the 34 s cost. Option:
split the synth user message into two content blocks — the corpus
(articles + cases, cacheable) and the instructions + advisory (short,
uncached). Set `cache_control: {"type":"ephemeral"}` on the corpus
block so the regen hits cache. Rough estimate: second attempt drops
from 34 s to ~15 s. Worth measuring before shipping.

### I5 — Add a corpus-quality filter at ingest for obvious OCR artefacts
In `src/jurist/ingest/caselaw.py` chunking stage, reject chunks matching
patterns like `\bplus l procentpunt\b`, `\b(\d)l(\d)\b` (digit-l-digit
context), or run a secondary pass with a cheap LLM over short
unigram-suspicious chunks. Not worth blocking on — but the demo has a
visible artefact right now.

### I6 — Tighten the decomposer concept list
Either enforce a JSON-Schema `enum` of allowed concept strings (hard —
requires an ontology), or post-filter against a known huurrecht term
list, or drop `concepts` entirely since downstream retrievers don't use
them. The frontend / trace surface might; check first.

### I7 — Add a real validator before v2
Even a small rule-based validator would catch classes of errors the
grounding check misses: duplicate IDs, label-ID disagreement,
conclusie/aanbeveling contradiction, quote length distribution. This
is pre-v2 scope per the design doc, but the stub hides real
regressions that would surface on non-locked questions.

### I8 — Make the eval harness a normal tool
`scripts/eval_m4_run.py` exists now; fold it into `uv run jurist-eval`
(an entry point in `pyproject.toml`) so the writeup style from this
doc can be reproduced on any branch, and so CI can optionally run it
gated on `RUN_E2E=1` nightly. Keeps `docs/evaluations/` comparable
across future milestones.

```bash

uv run python scripts/eval_m4_run.py
# -> out/m4-eval/{trace.jsonl, summary.json, answer.md}
```

Expected wall time: 90–180 s (the spread is L4). `answer.md` is
the surface to read for quality. `summary.json` is the surface for
comparing runs.

## 7. Summary

The pipeline works and produces a substantively correct, grounded
Dutch answer to the locked question. The closed-set citation grounding
— which was the entire point of M4 — is load-bearing on real traffic:
one of two runs hit the regen path and recovered silently. The
observability contract (every decision is an event) holds, once you
raise the buffer cap.

The top three things to fix before showing this to someone:
1. **L1 / I1** — raise `EventBuffer` default to 2000 so the trace panel
   actually shows the pre-synthesizer events.
2. **L2 / I2** — remove the misleading "synthesizer thinks out loud"
   claim from CLAUDE.md + the system prompt; set correct expectation.
3. **L4 / I4** — add cache_control on the synthesizer's corpus block so
   a regen doesn't double total wall time.

Everything else is polish.
